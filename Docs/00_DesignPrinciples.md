# 0. 설계 원칙

> Atlas_CyberDepot 아키텍처 설계안 v5 — §0. 전체 시스템에 공통 적용되는 원칙이므로, 다른 어떤 `Docs/` 문서를 작업하든 이 문서는 항상 함께 참고한다.

- 게임플레이 로직과 데이터 출력(CSV/MQTT/시각화)을 분리한다 → 전부 이벤트 버스 구독자로 둔다.
- 경로를 막는 대상은 전부 `AFactoryAgentBase` 파생 에이전트(정지한 고장 로봇, 이동 중인 NPC/플레이어)뿐이며, 회피는 Detour Crowd 한 시스템으로 처리한다. 별도의 정적 장애물 감지/캐싱 레이어는 두지 않는다. 단, NPC가 자신에게 배정된 정비 대상(`AssignedMaintenanceTarget`)에게 접근할 때는 해당 대상을 Crowd Avoidance 무시 목록에 등록해, 좁은 통로에서 고장 로봇이 동적 장애물로 인식되어도 접근이 차단되지 않도록 한다.
- 코스트 볼륨은 미리 배치된 풀(pool)을 재사용하고, 점유는 정수 카운터로, 해제는 최소 유지 시간(히스테리시스)을 거친다. **코스트 반영은 NavMesh 지오메트리를 직접 변경하지 않고, 에이전트별 `NavQueryFilter` 인스턴스의 AreaCost 조정만으로 처리한다** — 런타임 NavArea 클래스 스왑은 사용하지 않는다(타일 재빌드 회피).
- 리플레이는 로직 재실행이 아니라 기록된 상태(Transform/State)의 재생(Playback)이다.
- 업무가 없는 로봇은 바닥 아무 곳에나 멈추지 않고, 자기 타입 전용 대기 공간(`AIdleWaitingZone`)으로 이동해 파킹한다.
- 예방 정비(QuickCheck)는 로봇이 대기 공간에 실제로 파킹되어 있을 때만 수행 가능하다.
- 모든 이벤트/스냅샷 구조체는 `SchemaVersion` 필드를 갖는다.
- 작업 배정은 두 갈래로 나눈다: 아틀라스는 **거점(Zone)에 고정**되어 잔여 처리량이 0이 될 때까지 머무는 "거점 배정"을, 운송로봇은 **트립(왕복) 단위**로 작업을 풀(pull) 방식으로 받아가는 "트립 작업"을 수행한다. 단, 로봇이 위험 임계치(`IsMaintenanceDue() == true`)에 도달하고 대기실에 초기화된 교대 로봇이 있으면, 아틀라스는 잔여 처리량이 남아있어도 중도 핸드오프를 통해 교대할 수 있다.
- 물리적으로 단일 컨베이어인 트레이는 작업 구역에 아틀라스 1대+운송로봇 1대로 동시 접근을 제한하지만, 선반의 좌/우 구역은 아틀라스 1대만 거점으로 고정하되 운송로봇은 수 제한 없이 방문 가능하다. 슬롯 단위 충돌은 개별 예약으로만 방지한다.
- **고장 확률 롤은 이벤트(상태 전이 경계) 기반으로만 평가한다.** 운송로봇은 이동 직전마다, 아틀라스는 이동 직전 또는 `TransferItem` 직전에만 평가한다. 타이머 기반 폴링은 사용하지 않는다.
- **로봇이 고장나면 그 자리에서 즉시 멈추며(이동 불가), NPC/플레이어가 현장에서 수리한다.** 수리 완료 후 로봇은 고장 직전에 진행 중이던 작업(거점 배정·슬롯 예약 상태 포함)을 이어서 재개한다. 고장 발생 시 보유 중인 슬롯 예약이나 거점 배정은 해제하지 않는다. FullRepair는 유한 시간 내 항상 완료되므로, 예약에 별도의 타임아웃/리스(lease) 만료 로직을 두지 않는다(`14_OpenIssues.md` 참고).
- **물품 운반은 소켓 부착(Socket Attach) 방식으로 처리한다.** `HeldItem`/`PayloadItem`은 로봇 스켈레탈 메시의 전용 소켓에 `AttachToComponent`로 부착되며, 고장으로 애니메이션이 멈춰도 물품은 소켓에 그대로 유지된다. 물리 기반 홀딩은 사용하지 않는다.
- **물품 전달 애니메이션은 층별 몽타주(Floor-indexed Montage)를 사용한다.** 선반 층(FloorIndex 0·1·2)에 대응하는 몽타주 3종을 기본으로 하며, 소켓 부착/해제 타이밍은 AnimNotify로 트리거한다. IK는 선택적 폴리싱 항목으로 유보한다.
